# EP13.2（神之领域 / Encounter with the Unknown / Pre-Renewal）中文化方案调研

## 结论摘要
- **目前没有发现比 `najoast/rathena_npc_translate` 明显更完整、且能直接兼容当前 `rAthena master` 的现成 NPC 中文方案。**
- 当前仓库已经有一套更稳妥的落地方式：`tools/npc_cn_translate.py`，它不会覆盖旧脚本结构，而是只替换对话文本并输出 `GBK`。
- 如果目标是“尽可能完整的 EP13.2 中文体验”，**最佳路线不是换一套现成仓库**，而是采用：
  - `rAthena master` 作为服务端基线
  - `najoast` 作为 NPC 历史翻译底座
  - `ROClientFullCN` / `LeeClient` 作为客户端中文资源补充
  - 再对未覆盖 NPC 文本做增量补译

## 仓库内现状核实

### 1. 已有安全汉化工具
仓库内已存在 `tools/npc_cn_translate.py`，其设计目标非常明确：
- 保留当前 `rAthena` 的 NPC 坐标、精灵 ID、脚本逻辑
- 仅替换 `mes` / `select` / `caption` 等文本
- 输出 `GBK`
- 先备份英文文件

这正好规避了旧中文脚本直接覆盖导致的兼容性问题。

### 2. 当前汉化覆盖率
通过仓库内现有统计脚本核实，当前工作区 NPC 文本中文覆盖仍然偏低：

- NPC 文本文件总数：`1138`
- 含中文文件数：`142`
- 仍为英文文件数：`996`

按目录看，关键薄弱区很明显：
- `jobs`: `1/33`（3%）
- `quests`: `19/88`（21%）
- `warps`: `0/81`（0%）
- `pre-re`: `0/173`（0%）
- `re`: `50/475`（10%）
- `events`: `5/36`（13%）

这意味着：
- 现有方案足以作为“底座”，
- 但距离“完善中文翻译”还有明显差距，
- 尤其 `EP13.2 pre-re` 相关任务链、传送链、说明文本，不能指望现成仓库一次到位。

## 外部方案调研结果

### A. `najoast/rathena_npc_translate`
这是当前最有价值的 NPC 中文来源，但它本身也明确说明：
- 翻译来自当年的 `rAthenaCN / Pandas` 备份
- **不会跟进官方 rAthena NPC 改动**
- 只是历史备份，欢迎别人继续更新

这说明它的定位是：
- **可用的历史翻译底座**
- **不是实时维护的完整解决方案**

### B. `rAthenaCN/ROClientFullCN`
这是很有价值的中文资源库，但定位主要是：
- 完整客户端中文资源
- `iteminfo.lua`、`skillinfo`、系统 Lua、客户端补丁资源等

它适合解决：
- 客户端 UI / 物品 / 技能 / Lua 文本

但它**不等于完整服务端 NPC 脚本中文方案**。

### C. `PandasWS/LeeClient`
这个项目的定位也很清楚：
- 完整客户端整合
- 方便 GM 调试不同版本客户端
- 处理客户端资源与部分汉化工作流

它的价值主要在客户端侧，而不是直接提供一套可用的 `EP13.2 pre-re` NPC 服务端汉化。

### D. `RevoKey/rathena-zh`
调研结果看起来更像：
- 一个较旧的中文化 `rAthena` fork
- README 基本沿用老 `rAthena` 说明
- 没有足够证据证明它比 `najoast + 安全合并` 更完整、更现代、更兼容当前 master

对于当前目标，不建议把它当成主方案。

### E. `GreatWizard/rathena-intl`
这是个有意思的国际化尝试：
- 可以从 rAthena NPC 脚本中自动抽取待翻译文本
- 保存成 XLIFF 一类结构化翻译源

但从公开仓库信息看：
- 提交少
- 没有现成中文语料成果
- 更像“翻译流程框架”，不是现成中文包

它适合参考思路，不适合作为立即可落地的主方案。

## 推荐方案

## 方案选择
**推荐：混合增量方案**

### 1. 服务端主线
- 继续使用 `rAthena master`
- 通过配置运行 `Pre-Renewal`
- 保留当前 EP13.2 相关脚本与主线兼容性

### 2. NPC 中文底座
- 继续使用 `najoast` 作为最好的现成 NPC 中文来源
- 但只能通过 `tools/npc_cn_translate.py` 这种“只替换文本”的安全方式合并

### 3. 客户端中文资源
- 接入 `rAthenaCN/ROClientFullCN`
- 如需完整客户端整合与补丁管理，可参考 `PandasWS/LeeClient`

### 4. 补全未覆盖翻译
- 先补最影响体验的目录：
  1. `quests`
  2. `jobs`
  3. `warps`
  4. `pre-re`
  5. `events`
  6. `EP13.2` 直接相关脚本（如 `npc/quests/quests_13_2.txt`、对应地图/传送/提示链）
- 建议建立“抽取 -> 翻译 -> 回填 -> GBK 校验 -> 游戏内验证”的增量流程

## 为什么不建议换方案
- **没有发现现成、更完整、持续更新、直接兼容 `rAthena master` 的中文 NPC 仓库。**
- 旧中文 fork 最大的问题不是“没中文”，而是“会破坏当前脚本兼容性”。
- 所以正确问题不是“找另一份完整旧汉化”，而是“如何基于当前 master 安全地持续补齐中文”。

## 给用户的实际建议

### 如果目标是“尽快上线一个可玩的 EP13.2 中文服”
建议：
1. `rAthena master + Pre-Renewal` 作为核心
2. 应用当前仓库的安全 NPC 汉化工具
3. 同步接入 `ROClientFullCN`
4. 把 13.2 主线任务、传送、关键城市/野外说明优先补齐

### 如果目标是“做一套长期维护的完整中文化基线”
建议：
1. 保留现有安全合并工具
2. 新增“待翻译文本抽取与覆盖率统计”
3. 建立 EP13.2 / pre-re 专项校验清单
4. 后续持续补译并用游戏内回归验证

## 最终判断
- **更好的“现成替代仓库”：没有明确找到。**
- **更好的“整体方案”：有。**
- 这个更好的方案不是“换源”，而是：
  - 继续使用 `rAthena master`
  - 以 `najoast` 为底座
  - 结合 `ROClientFullCN / LeeClient`
  - 再通过增量翻译流程补足 `EP13.2 pre-re` 缺口

这也是当前最稳、最现实、最容易长期维护的路线。
