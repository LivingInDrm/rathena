# 国内在运营的中文 Pre-Renewal RO 私服调研

## 结论摘要
- **国内/中文圈仍然存在正在运营的 `Pre-Renewal / R前二转 / 进阶二转 / 怀旧` RO 私服。**
- **这些服的公开首页通常不会直接披露“服务端到底是 rAthena、Hercules 还是自研分支”**，因此不能对单个服做武断判断。
- 从可公开验证的中文生态资料看，**最可信的判断是：主流中文 Pre-Renewal 服大多建立在 `rAthena / eAthena` 血缘分支之上，再叠加大量私有改动；完整中文化通常是“服务端 NPC 文本 + 客户端中文资源 + 启动器/补丁 + 社群说明文档”四层组合，不是单靠一个仓库解决。**

## 调研边界
- 本调研只记录**公开可验证**的信息。
- 对“服务端方案”的判断分两类：
  1. **直接证据**：官网/仓库/页面明确写出 `rAthena`、`Hercules` 等。
  2. **生态推断**：运营服未公开，但结合中文 RO 社区常用工具链、资源仓库与版本特征做保守判断。
- 结论优先服从 `docs/dev/architecture.md`：如果我们要落地自己的中文 Pre-Renewal 方案，应优先选择 **`rAthena master + 安全合并中文 + GBK 输出 + 客户端中文资源补齐`** 的路线。

## 这些服大概率使用什么服务端方案

## 可直接确认的部分
- 这批运营服的公开首页/列表页，**基本不直接写服务端内核**。
- 因此，针对 `Rohome`、`海诺仙境`、`海棠仙境`、`英雄仙境` 等具体服，**我们没有拿到足以“点名确认”的公开直接证据**。

## 生态层面的强证据

### A. 中文 RO 技术生态明显偏 `rAthena`
- `rathena/rathena` 仍是最活跃、最公开、最标准化的开源 RO 服务端之一。
- 中文相关资源几乎都围绕它展开：
  - `najoast/rathena_npc_translate`
  - `rAthenaCN/ROClientFullCN`
  - `PandasWS/LeeClient`
- 其中：
  - `najoast` 明确说明翻译来自当年的 `rAthenaCN / Pandas` 备份；
  - `ROClientFullCN` 提供完整中文客户端资源；
  - `LeeClient` 明确面向中文 GM/客户端整合工作流。

### B. 中文站点直接出现 `rAthena` 品牌
- `betterra.tw` 的公开检索摘要直接写有 `rAthena - GM平台`。
- 虽然站点被 Cloudflare 验证挡住，无法完整抓取正文，但这至少说明：
  - 中文圈公开提供的模拟器/GM 平台，**仍把 `rAthena` 当成核心品牌资产**；
  - `rAthena` 在中文私服技术生态中的可见度，显著高于 `Hercules`。

## 保守判断
- **最稳妥的结论不是“这些服都一定是原版 rAthena”**。
- 更准确的说法应是：
  - **中文 Pre-Renewal 服大概率以 `rAthena / eAthena` 血缘代码为基础；**
  - **实际运营时通常会加入大量私有功能、技能平衡、倍率系统、挂机系统、家园系统、商城/活动脚本，所以更像“rAthena 分支/魔改服端”，而不是公开仓库原样部署。**
- 目前**没有公开证据证明 `Hercules` 在中文 Pre-Renewal 运营服里占主导地位**。
- 因此如果我们要做自己的中文 Pre-Renewal 服，**首选继续以 `rAthena` 体系为基线**，而不是为了“贴近市场”改投 Hercules。

## 它们如何实现完整中文化

## 结论先行
这些服的“完整中文化”通常不是单一来源完成，而是下面四层叠加：

1. **服务端 NPC/任务脚本文本中文化**
2. **客户端资源中文化**
3. **客户端补丁/启动器分发**
4. **站外中文说明体系（官网、QQ群、攻略页、Wiki）**

## 1. 服务端 NPC 文本层
- 公开可验证的最佳中文来源仍然是 `najoast/rathena_npc_translate`。
- 该仓库自己也说明：
  - 它是历史备份；
  - 不会持续跟进官方 rAthena 全量 NPC 改动；
  - 来源是早期 `rAthenaCN / Pandas` 的翻译资产。
- 这意味着中文服常见做法很可能是：
  - 以旧中文 NPC 语料做底座；
  - 再按自己服的版本与自定义内容继续补译、重写、删改。

### 对我们最有价值的启示
- **不能直接覆盖 `npc/` 原文件。**
- 正确方式应当是：
  - 保留当前英文脚本结构、坐标、精灵 ID、逻辑；
  - 仅替换 `mes`、`select`、`caption` 等文本；
  - 输出 `GBK`，保证老客户端显示正常。
- 这与当前仓库的 `tools/npc_cn_translate.py` 设计完全一致。

## 2. 客户端中文资源层
- `ROClientFullCN` 直接展示了中文客户端资源的典型构成：
  - `System`
  - `data`
  - `iteminfo` / `skillinfo` / Lua 文件
  - 已补丁的 `Ragexe`
  - 若干 DLL、GRF、按钮/UI 图档
- `LeeClient` 则更像一个**客户端整合与补丁工作流**：
  - 汇总官方图档；
  - 维护汉化覆盖率；
  - 适配新版/旧版客户端；
  - 做按钮、描述、资源合并。

### 这解释了“完整中文化”为何不是只翻 NPC
- 光有服务端中文 NPC，不会自动得到：
  - 物品名中文
  - 技能名中文
  - 系统 UI 中文
  - msgstringtable / Lua / 按钮 / 登录器 中文
- 所以完整中文体验，**必须配客户端中文资源包**。

## 3. 启动器 / 补丁分发层
- `海诺仙境` 这类站点通常会提供：
  - 下载中心
  - 注册页
  - 帮助页
  - 自定义 Wiki / 新手指南
- 这说明真实运营中，中文化不是“一个源码仓库”交付，而是：
  - 一个打包好的客户端；
  - 搭配补丁、说明、注册链接、群入口一起交付。

## 4. 站外中文说明体系
- 很多运营服的中文体验，并不完全依赖游戏内文本。
- 例如 `海诺仙境` 把以下内容放在站外中文页面中：
  - 新手练级/转职/转生指引
  - 家园系统
  - 装备附魔
  - 每日签到
  - 副本说明
  - 职业攻略
- 这说明：**“完整中文化”在运营视角上，往往是“游戏内中文 + 游戏外中文知识库”的组合。**

## 对本仓库的直接建议

### 1. 服务端方案
- 保持 `rAthena master` 为主线。
- 继续使用 `Pre-Renewal` 模式。
- 不为了模仿市面服而替换到未知中文分支。

### 2. NPC 中文化
- 继续以 `najoast` 作为历史翻译底座。
- 通过仓库内安全合并工具处理，不直接覆盖原脚本。
- 严格坚持 `GBK` 输出。

### 3. 客户端中文化
- 引入 `ROClientFullCN` 作为客户端中文资源基线。
- 如需要更完整的客户端整合/补丁能力，可参考 `LeeClient` 的工作流。

### 4. 体验补齐
- 优先补齐最影响“像中文服”的内容：
  1. `quests`
  2. `jobs`
  3. `warps`
  4. `pre-re`
  5. `events`
  6. 13.1/13.2 主线相关任务链
- 同时补游戏官网/帮助页/FAQ 文案，而不是只盯着游戏内对话。

## 最终判断
- **“国内是否还有在运营的中文 Pre-Renewal RO 私服？”——有，而且不止一个。**
- **“它们公开使用什么服务端方案？”——多数不会公开写明。**
- **“从公开生态最可信的推断是什么？”——大概率以 `rAthena / eAthena` 血缘分支为底，再做长期私有魔改。**
- **“它们如何实现完整中文化？”——服务端 NPC 中文 + 客户端中文资源 + 启动器/补丁 + 中文站外帮助体系，多层叠加。**
- 对本仓库而言，这进一步支持当前设计方向：
  - **不要追求一份‘现成完整中文服端’；**
  - **而要基于当前 `rAthena master` 做可持续的安全中文化流水线。**
